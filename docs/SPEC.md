## 1. Видение

Кросс-платформенное приложение (Android/iOS, web как бонус) с двумя продуктовыми режимами:

1. **Grip/community (`grip`)** — режим для всех обычных пользователей: свободная тренировка без
   программ, учёт эспандеров и grip-подходов, личные рекорды/аналитика и общий лидерборд.
2. **Private athlete (`full`/`admin`)** — закрытый режим только для Сергея и Марии: программы,
   ИИ-импорты, ИИ-коуч, OURA и цикл. `admin` дополнительно модерирует лидерборд.

Запись в зале должна быть **быстрой и не зависеть от качества сети**. ИИ — под жёсткими
per-user лимитами и никогда не доступен роли `grip`.

**Принципы (прод-планка):**
- Качество прод-уровня: не «очередная тренинг-апка». Каждая кнопка → понятный граф запросов;
  лишней сети нет; данные не теряются.
- **Offline-first для записи тренировки** (см. §4) — ключевой инвариант, базовый durable-контур
  реализован; оставшиеся риски синхронизации закрываем до расширения функционала.
- Роль хранится только в `user_roles`: новые аккаунты получают `grip`; `full/admin` выдаются
  вручную только Сергею и Марии. Скрытый таб не является защитой — платные/приватные функции
  проверяют роль на сервере.
- ИИ и внешние API — **per-user лимиты** + глобальный kill-switch; ключи только на сервере.
- Метрики (тоннаж/1ПМ/PR/тренды) считаются **в SQL/RPC**, не в клиенте и не в ИИ.
- Файлы (xlsx/docx/pdf/видео) и аудио **не хранятся**: распарсил → структура в БД → исходник выкинут.
- Воспроизводимость: всё в облаке + git (`clone` → `npm i` → `.env` → запуск).

- 📝 Онбординг должен собирать `bodyweight`/`height_cm` (нужны коучу/метрикам) и заканчиваться

### 3.2 Каталог упражнений + таксономия — ✅
- `name_en`/`name_uk`/`aliases[]` (оба языка + сленг), `cluster` (upper/lower/full/core),
  `category` (general/weightlifting/gymnastics/crossfit/armwrestling/kettlebell), `is_global`.
- Пикер: typeahead + группировка по кластеру; «часто используемые» сверху; создание кастомного
  (анти-спам через триггер `exercise_daily_cap`).
- 📝 Дедуп/мердж кастомных (импорт + ручные плодят дубли — мерджили вручную миграцией).
- 📦 Каталог попадает целиком в промпт импорта (§3.5) — при росте каталога дорожает у каждого
  импорта; перейти на поиск-кандидатов.

### 3.3 Запись тренировки (ядро) — ✅ offline-first, ⚠️ hardening
- Старт из программы (префилл) или «с нуля»; подходы: вес (kg/lb→канон. кг), повторы/секунды
  (per-set переключение reps⟷time), RPE (зоны 🟢1-4/🟡5-7/🔴8-10), сторона (ліва/права/обидві=×2
  объём), читинг, эспандер для грип-упражнений (`sets.meta`).
- Кластеры (суперсет/EMOM/E2MOM/круг) раунд-за-раундом; авто-отдых = разрыв с прошлым «✓»
  (только в живой сессии); таймеры тренировки/отдыха/EMOM; сворачивание карточек; resume-баннер.
- ✅ Старт/подходы/структура/завершение используют клиентские UUID, optimistic cache и
  persisted paused mutations с восстановлением после перезапуска и retry на реконнекте.
- ⚠️ До beta закрыть: единственную активную тренировку на пользователя; порядок конкурентных
  обновлений одного подхода; явное обнуление пустого `sets.meta`; query error/retry UX.

### 3.4 Программы — ✅ редактор, 🟡 развитие

- ✅ Ручное создание пустой программы и полный редактор.

## 4. Offline-first (приоритет №1 — переписываем ядро записи)

**Текущее состояние.** Этап 1 реализован без отдельного sync-движка:

1. Клиентские UUID для workout/workout_exercise/set и идемпотентные upsert.
2. `persistQueryClient` + AsyncStorage: кэш и paused mutations переживают перезапуск.
3. `onlineManager` связан с NetInfo; durable defaults регистрируются по `mutationKey`.
4. Горячий путь тренировки оптимистичен; ошибки видны через `SyncStatus`, есть ручной retry.
5. Платный ИИ и неидемпотентные community-записи намеренно не получают глобальный retry.

**Оставшийся hardening:**

1. Не допускать больше одной незавершённой тренировки локально и уникальным partial index в БД.
2. Сериализовать обновления каждого подхода, чтобы старый ответ не перетёр более новое значение.
3. Тестировать сценарий offline → kill/restart → reconnect → replay без дублей и потерь.
4. Явно разделить fresh/stale/offline/error состояния на экранах чтения.

**Этап 2 (если масштаб потребует настоящего sync-движка):** оценить **PowerSync/ElectricSQL**
(first-class с Supabase) для двунаправленного offline-first. 💰 Это доп. сервис/стоимость и
сложность — берём только если replay-модель этапа 1 упрётся. Решение по факту, не авансом.

**Совместимость с Expo Go (SDK 54):** все зависимости этапа 1 — JS-only или встроены в Expo Go
(NetInfo, expo-crypto). Перед установкой сверять по доке v54, не тянуть нативное, ломающее Expo Go.

## 5. Данные (Postgres / Supabase, RLS обязателен)

Реальная схема (упрощённо; полная — в `supabase/migrations/`):
```
profile(user_id, display_name, language, units, gender, gender_self, bodyweight, height_cm,
        disciplines[], track_cycle, oura_connected, onboarded_at)
exercises(id, owner_id|null, name_en, name_uk, aliases[], cluster, category, is_global, metric, log_kind)
programs / program_blocks / program_exercises / program_sets        -- план (целевые значения)
workouts / workout_exercises / sets                                  -- факт (sets.meta = side/cheat/gripper_id/set_type)
grippers(id, owner_id|null, brand, name, rgc, rgc_unit, is_global)
health_snapshots(user_id, date, readiness, sleep_*, hrv, rhr, temp, …)
cycle_periods(user_id, start_date)
coach_facts(user_id, kind, content)
ai_threads / ai_messages
ai_usage(user_id, period_type, period_key, intent, tokens_in, tokens_out, requests, cost_estimate)
ai_limits(singleton)  ai_model_routes(intent → provider/model/price)
private.oura_tokens(user_id, access_token, refresh_token, expires_at)  -- вне публичной схемы
```
- RLS: каждый видит только свои строки; вложенные — через `exists(...)` на родителя. Edge Functions
  под service_role, но всегда скоупят по `userId`.
- 📦 **Хранение на масштабе:** Supabase Free = 500 МБ — хватит на десятки юзеров, но при 100–1000
  и истории подходов/`health_snapshots` (365 дн/юзер) упрёмся → план Pro ($25/мес, 8 ГБ) и/или
  ретеншн/агрегация старых `ai_messages`/`health_snapshots`. Заложить индексы под per-user выборки.

## 6. ИИ-гейтвей, косты и лимиты

**Гейтвей** (`_shared/ai/gateway.ts` `runIntent`): `ai_budget_check` → роут (`ai_model_routes`:
интент→провайдер/модель/цены) → адаптер → `ai_record_usage` (токены+кост). Ключи только в Edge env.

**Модели (Claude, $/1М ток.):** Haiku 4.5 $1/$5 (рутина/классификатор) · Sonnet 4.6 $3/$15 (коуч/
разбор/импорт) · Opus 4.8 $5/$25 (опц. глубокий анализ). Мультипровайдер → ставим дешевле там, где
не теряем качество. Системный промпт — под prompt caching.

**Лимиты (`ai_limits`, дефолты):** общий `month_cost_cap=$30`, персональный
`user_month_cost_cap=$20`, `day_request_cap=60`/юзер, `day_token_cap=200000`/юзер, `kill_switch`.
Сначала проверяется лимит конкретного пользователя, затем общий аварийный потолок. Роль
`grip` блокируется до вызова модели.

⚠️ Перед масштабом/монетизацией нужны entitlements по тарифу и кламп/санитайз стоимости в
`ai_record_usage`; общий $30 остаётся жёстким требованием текущего private-режима.

💰 **Оценка костов:** на единицах юзеров — $5–15/мес. На 100–1000 — масштабируется с числом
активных; держим через тиринг моделей (Free→Haiku, Pro→Sonnet), компактный контекст коуча
(треды + сводка, не вся история), prompt caching и per-user квоты.

## 10. Порядок работ (прод, с учётом масштаба 100–1000)

**P0 — целостность данных и доверие к записи:**
1. Единственная активная тренировка: клиентский resume guard + partial unique index.
2. Сериализация обновлений одного подхода и корректное обнуление `sets.meta`.
3. Транзакционный commit AI-импортов без частичных program/workout.
4. Error/retry/offline состояния вместо ложных пустых экранов.
5. Автотесты критических pure-функций и offline mutation contracts; затем integration/E2E smoke.

**Качество/долги:**
6. Стартовый каталог и экран активной программы для private-режима.
7. Вернуть edit/delete/history дат цикла; OURA OAuth + авто-синк.
8. Exercise drilldown в аналитике; точные ошибки коуча.
9. Дедуп кастомных упражнений; защита `meta.gripper_id`; preview импорта.

**Релиз:** password reset, export/privacy/terms; затем iOS bundle id/TestFlight по явной команде.

**Косметика/UX:** нумерация тренировок при >30; дискаверабилити «імпорт минулого»/«старт з нуля»;
онбординг-CTA; сбор bodyweight/height.
