## 0. Главное, что надо знать до сценариев (архитектурная правда)

Эти факты «протекают» во все сценарии ниже — выношу один раз.

### 0.1 ❗Никакого offline-first / sqlite на самом деле нет
`AGENTS.md` и `SPEC.md` обещают: «expo-sqlite — offline-first запись + очередь синка в
Supabase». **В коде этого нет.** `expo-sqlite` стоит в зависимостях и в плагинах
`app.json`, но ни один экран/`src/lib/db/*` его не использует. Весь доступ к данным — это
**прямые online-вызовы Supabase через TanStack Query** (`src/lib/supabase.ts` + `@/lib/db/*`).

Последствия:
- Нет сети → нельзя залогировать ни одного подхода, открыть тренировку, увидеть аналитику.
  Каждый тап «✓ сделано» = сетевой `UPDATE`.
- Это и есть ответ на твою боль «не могу вести тренировки, не развернув эту шляпу»:
  приложение **полностью зависит от живого Supabase + интернета**, а в деве — ещё и от Metro
  в одной сети. После EAS-сборки (раздел 17) Metro уйдёт, но **зависимость от интернета
  останется** — в зале с плохим сигналом подходы будут терять/не сохранять.
- **Решение** (выбор архитектора): либо честно внедрить offline-кэш (sqlite/MMKV + очередь
  мутаций + `onlineManager`/`persistQueryClient` у TanStack), либо убрать обещание из ТЗ и
  принять online-only как сознательное ограничение MVP. Сейчас — «ни то, ни сё»: обещано, но
  не сделано, и это самый крупный технический долг.

### 0.2 Хранилище данных и RLS
- Источник правды — **Postgres в Supabase**. Таблицы: `profile`, `programs`/`program_blocks`/
  `program_exercises`/`program_sets`, `workouts`/`workout_exercises`/`sets`, `exercises`,
  `grippers`, `personal_records`, `health_snapshots`, `cycle_periods`, `coach_facts`,
  `ai_threads`/`ai_messages`, `ai_usage`/`ai_limits`/`ai_model_routes`, `private.oura_tokens`.
- **RLS включён** на пользовательских таблицах — каждый видит только свои строки (`user_id =
  auth.uid()`), вложенные таблицы — через `exists(...)` на родителя. Edge Functions работают
  под `service_role` (обход RLS), но **всегда скоупят по `userId`** вручную.

### 0.3 ИИ-гейтвей и бюджет (косты)
- Любой вызов модели идёт через `supabase/functions/_shared/ai/gateway.ts` → `runIntent()`:
  **(1)** `ai_budget_check` → **(2)** роут из таблицы `ai_model_routes` (интент→провайдер/модель/
  цены) → **(3)** адаптер (Claude/OpenAI/Gemini) → **(4)** `ai_record_usage` (запись токенов+кост).
- Лимиты (`ai_limits`, дефолты из миграции): `month_cost_cap = $30`, `day_request_cap = 60`/юзера,
  `day_token_cap = 200000`/юзера, есть `kill_switch`.
- ❗**Архитектурная мина:** `month_cost_cap` суммируется **по всем юзерам без фильтра по
  user_id** (`sum(cost_estimate) where period_type='month'`). День-34 это уже взорвал: один битый
  `ai_usage` (STT записал ~$89M из-за эпохи-в-мс) упёр глобальный потолок и **положил весь ИИ для
  обоих юзеров**. Дневные капы — по юзеру, месячный кост-кап — глобальный. Это значит: один
  пользователь (или один баг) может заблокировать ИИ всем. Под монетизацию (раздел 16) это надо
  переделать на per-user бюджет.
- Ключи провайдеров и OURA PAT — **только на сервере** (Edge env / `private.oura_tokens`),
  на клиент не попадают. ✅ соответствует правилам.

### 0.4 Верификация
- Тестов нет. Единственная проверка — `npx tsc --noEmit`. Нельзя говорить «протестировано» —
  только «типы сходятся». Это повышает цену любого рефактора (раздел 17).

### 0.5 Навигация (IA)
Гейт `src/app/index.tsx`: нет сессии → `/auth`; есть, но нет `onboarded_at` → `/onboarding`;
иначе → `/workouts`. Табы (`src/app/(tabs)/_layout.tsx`): **Тренировки · Програми · Коуч ·
Аналітика · Здоров'я · Акаунт** (6 табов). Экраны вне табов: `auth`, `onboarding`,
`workout/[id]`, `summary/[id]`, `program/[id]`, `exercises`, `grippers`.

## 1. Первый запуск → регистрация → онбординг

**По-человечески.** Открыл приложение → экран входа (e-mail + пароль, «Увійти» / «Зареєструватися»).
Зарегистрировался → попал на онбординг: язык (en/uk), единицы (kg/lb), пол (+своя формулировка
для «other», +тумблер «вести цикл» для female) → «Далі» → главный экран «Тренування».

**Под капотом.**
- `src/app/auth.tsx`: `supabase.auth.signUp/signInWithPassword`. Если у проекта включено
  подтверждение почты — `signUp` не вернёт сессию, показывается «перевір пошту».
- `src/app/onboarding.tsx`: `profile.upsert({language, units, gender, gender_self, track_cycle,
  onboarded_at})` — **best-effort** (обёрнут в try/catch, ошибка не блокирует переход) + дубль во
  `AsyncStorage` (`app.language`, `app.onboarded`) как оффлайн-фолбэк для гейта.

**Риски / ботлнеки / что чинить.**
- ❗**Нет восстановления пароля** (forgot/reset) и нет OAuth (Google/Apple). Для публичного
  релиза Apple обычно **требует Sign in with Apple**, если есть сторонний логин. Это «не допилено А».
- ❗**Нет подтверждения сложности пароля / валидации e-mail** на клиенте — только то, что вернёт
  Supabase. Сырой `error.message` от Supabase показывается пользователю как есть (англ., техничный).
- ❗**Онбординг не собирает данные, которые потом нужны коучу и метрикам:** `bodyweight`,
  `height_cm`, дисциплины, дата рождения. Коуч умеет читать `bodyweight`/`height_cm`/`disciplines`
  (см. `get_profile`), но онбординг их не спрашивает → у новичка они пустые. Это твои «Ц, Г».
- **Нет шага «подвести к импорту программы».** Ты сам сказал — пользователя надо аккуратно
  довести до вкладки «Програми». Сейчас онбординг просто кидает на пустой список тренировок без
  единой подсказки «с чего начать». Стоит добавить финальный слайд-CTA «Імпортувати програму» /
  «Записати тренування» / «Підключити OURA».
- **best-effort профиль** = если `upsert` упал (нет сети), `onboarded_at` в БД не записался, но
  локальный флаг стоит → на другом устройстве человек снова увидит онбординг. Мелочь, но след
  «мультидевайс» (это твой кейс).
- Подтверждение почты не обрабатывается deep-link'ом (`detectSessionInUrl: false`) — после клика
  в письме на вебе сессия не подхватится автоматически.

## 2. Повторный вход / запуск с сессией

**По-человечески.** Открыл приложение — если уже логинился, сразу попадаешь на «Тренування»
(или дорегистрируешь онбординг).

**Под капотом.** `AuthProvider` (`src/lib/auth/auth-context`) держит сессию (persist в
AsyncStorage, autoRefresh). `index.tsx` спрашивает `profile.onboarded_at` (источник правды —
БД, синкается между устройствами) и роутит. Язык/единицы поднимаются ещё в `_layout.tsx` до
рендера маршрутов (иначе при релоаде на `/workouts` сбрасывались бы на en/kg).

**Риски / что чинить.**
- Гейт делает сетевой запрос профиля на каждом холодном старте; нет сети → ловит catch →
  фолбэк на локальный флаг (ок). Но при медленной сети — лишний спиннер.
- `(tabs)/_layout.tsx` повторно гардит сессию (deep-link мимо гейта) — это правильно.

## 3. Импорт программы через ИИ

**По-человечески.** Вкладка «Програми» → «Імпортувати програму» → вставляешь текст плана
(хоть из мессенджера, хоть свой) → «Імпортувати». Крутилка «розкладаю програму» → открывается
готовая программа с блоками/упражнениями/подходами.

**Под капотом.**
- `programs.tsx` → `importProgram(text)` → `supabase.functions.invoke('program-import')`.
- `program-import/index.ts`: auth → читает `profile.units` (числа в тексте трактует в единицах
  юзера, хранит канонически в **кг**) → тянет каталог упражнений (`is_global` + личные) →
  `runIntent('program_import', {json:true, maxTokens:4096})` с большим system-промптом (блоки
  rounds/EMOM/AMRAP/superset/single, правила нейминга, развёртка мультипликаторов `4x8`→4 подхода,
  грипперы как модель-а-не-вес, и т.д.).
- Парсит JSON → создаёт `programs` (source=`ai_import`) → по блокам создаёт `program_blocks` →
  `resolveExerciseId()` матчит каждое упражнение: доверяет `catalog_index` **только если имена
  реально похожи** (`namesResemble`), иначе ilike-поиск, иначе заводит **кастомное упражнение**
  пользователя (`is_global:false`). Подходы пишутся в `program_sets`. Повтор того же снаряда подряд
  дописывается **подходами в предыдущее упражнение**, а не плодит дубль.

**Риски / ботлнеки / что чинить.**
- **Стоимость и латентность:** один импорт = до 4096 выходных токенов + весь каталог в каждом
  промпте. Каталог растёт → промпт дорожает линейно у **каждого** импорта. При росте каталога
  стоит перейти на поиск-кандидатов (top-N по эмбеддингу/триграммам) вместо «весь каталог в промпт».
- ❗**Глобальный кост-кап (0.3):** упёрся месячный потолок — импорт падает `budget_exceeded`
  (429) у всех. У клиента это маппится в `programs.errBudget`.
- **`parse_failed` / `no_exercises`** — если модель вернула невалидный JSON. Сейчас просто ошибка
  юзеру; нет ретрая/ремонта JSON. Можно добавить один авто-ретрай с «верни строго JSON».
- **Матчинг — главный источник «мусора»:** часть упражнений уедет в кастомные (`exercise_id=null`),
  в программе они помечаются жёлтым `unmatched`. Это сознательно (лучше null, чем кривой матч),
  но плодит дубли каталога между импортами. Стоит дедуп кастомных по нормализованному имени.
- **Нет превью перед сохранением:** импорт сразу пишет программу в БД. Если разложилось криво,
  у юзера остаётся мусорная программа (правится руками, см. сценарий 4). UX лучше: показать
  разобранное → «зберегти / переробити».
- **Лимит 12000 символов** на вход (`text.slice(0,12000)`) — молча обрежет длинный план.

## 4. Просмотр и редактирование программы

**По-человечески.** Тап по программе → блоки и упражнения. Кнопка «Редагувати»:
переименовать программу/упражнения, добавить/удалить/изменить подходы, переставить упражнения
↑↓, удалить программу. Внизу — «Почати» (старт тренировки).

**Под капотом.** `program/[id].tsx` + `db/programs.ts`. Всё — точечные Supabase-мутации с
`invalidateQueries(['program', id])` после каждой. Вес показывается/редактируется в выбранной
единице, в БД летит через `toKg`. `groupProgram()` решает, что показать общей шапкой (настоящий
кластер — суперсет/EMOM/круг), а что — отдельными карточками (`single`-блок, даже если импорт
склеил несколько движений в один блок).

**Риски / что чинить.**
- **N запросов на перестановку** (`reorderProgramExercises` — `Promise.all` по одному update на
  упражнение). Для длинных программ это пачка запросов; не критично, но не атомарно (упадёт
  половина — порядок поедет).
- `exercise_id == null` (unmatched) упражнения **нельзя добавить в тренировку** (см. 5) — а в
  редакторе программы это видно только жёлтой меткой. Пользователь не понимает, почему упражнение
  «не стартует». Стоит дать прямо в редакторе «прив'язати до каталогу».
- Правки идут по `onEndEditing` (потеря фокуса). На вебе/в некоторых кейсах фокус не теряется →
  значение может не сохраниться. (Та же модель и в тренировке — см. 5.)

## 5. Старт и ведение тренировки (ядро продукта)

**По-человечески.** «Почати» в программе → экран тренировки с префилл-упражнениями. Вводишь
вес/повторы (или секунды для статики), RPE (зоны лёгко/средне/тяжело), сторона (ліва/права/обидві),
читинг, эспандер для грип-упражнений. Тап «✓» — подход сделан, считается отдых. Можно добавлять
подходы/упражнения/кастомные на лету, сворачивать карточки, завершать упражнение/блок. Сверху —
таймер тренировки и «відпочинок зараз». «Завершити» → экран-саммари.

**Под капотом.**
- `program/[id].tsx` → `startWorkoutFromProgram()`: создаёт `workouts`, **пакетно** (2 запроса)
  вставляет `workout_exercises` + `sets` из плана (вес из кг → в активную единицу, EMOM-круги
  вычисляются). Unmatched-упражнения (`exercise_id=null`) **пропускаются** — в тренировку не попадают.
- `workout/[id].tsx` — самый большой экран. `getWorkoutDetail` тянет всё дерево одним select
  (`workout_exercises(*, exercise:exercises(*), sets(*))`). Каждый тап = отдельная мутация
  (`updateSet`/`setSetLogged`/`addSet`/...) + invalidate.
- **Отдых:** при отметке «сделано» в **живой** тренировке `rest_sec = now − время последнего
  сделанного подхода` (anchor). При правке **завершённой** (`ended_at != null`) — не пересчитывает
  (иначе «бешеное время отдыха», фикс дня-32).
- **Метрики саммари** считаются локально (`workoutStats`): «обидві» → ×2 на объём; невыполненные
  подходы не считаются.
- `finishWorkout` ставит `ended_at` только если был null (повторное «Завершити» не раздувает
  длительность).

**Риски / ботлнеки / что чинить.**
- ❗**Online-only прямо в зале (см. 0.1).** Это самый болезненный риск ядра: каждый «✓» —
  сетевой UPDATE. Нет сети/лёг Supabase → тренировку не записать. Это №1 на починку.
- **Гонки/потеря ввода на `onEndEditing`:** значение, набранное без снятия фокуса, спасается
  только хаком «перед ✓ вызываем save()». Поля веса/повторов без явного blur могут не сохраниться.
  Стоит перейти на debounce-autosave или сохранять по `onChangeText` с дебаунсом.
- **Нет оптимистичных апдейтов:** после каждой мутации — `invalidate` → повторный полный
  refetch дерева тренировки. На большой тренировке это заметные перезагрузки и «прыжки» UI.
  Напрашиваются optimistic updates TanStack Query.
- **`updateSetMut` без onSuccess-invalidate** (в отличие от остальных) — значит правка значения
  подхода не рефетчит; полагается на локальный стейт `SetRow`. Рассинхрон возможен при кластерах.
- **Нет автopause/таймера отдыха с уведомлением** — «відпочинок» только показывает счётчик, не
  пушит «пора». Для силовой это ожидаемая фича.
- **`exercise_daily_cap`** (триггер БД) ограничивает создание кастомных упражнений в день
  (анти-спам) — при упоре юзер видит `workout.customCapped`. Ок, но лимит «невидимый» до удара.
- **Нет «черновика»/восстановления при краше** — стейт свёрнутости и т.п. в памяти; данные —
  в БД (это плюс), но незаписанный ввод теряется.

## 6. Свободная тренировка (без программы)

**По-человечески.** На «Тренування» — баннер активной сессии (resume) или старт «с нуля»,
добавляешь упражнения из каталога/поиском, заводишь кастомные.

**Под капотом.** `(tabs)/workouts.tsx` — список последних 30 тренировок (`listWorkouts`),
пульс-строка (последняя/за неделю/тоннаж), resume-баннер незавершённой. Пикер упражнений
(`ExercisePicker` в `workout/[id].tsx`): поиск пробивает весь каталог; просмотр — база +
включённые дисциплины + свои; «часто используемые» сверху (`getRecentExercises` по 25 последним
тренировкам). Выбор упражнения из не-включённой дисциплины **тихо включает её словник**.

**Риски / что чинить.**
- `getRecentExercises` тянет 25 тренировок со всем деревом ради частотности — дороговато,
  можно вынести в RPC/материализованное.
- Чип `#N` номера тренировки «стабилен пока загружено ≤30» (комментарий в коде) — после 30
  тренировок нумерация поедет. Косметика, но заметят.
- Старт «с нуля» как явная кнопка/флоу слабо выражен — основной путь подразумевает программу.

## 7. Кастомные упражнения, дисциплины, грипперы/эспандеры

**По-человечески.**
- *Кастомное упражнение:* в поиске пикера нет нужного → «＋ Створити "…"».
- *Дисципліни:* в «Акаунт» включаешь армрестлинг/гирі/… — это расширяет видимый словник.
- *Грипперы:* «Акаунт → Мої еспандери» — каталог брендов (Heavy Grips, CoC…) + личные; в
  грип-упражнении выбираешь модель + вид установки (set_type), пишется в `sets.meta.gripper_id`.

**Под капотом.** `db/exercises.ts` (`createCustomExercise` с триггер-капом, `getDisciplines`/
`setDisciplines` в `profile.disciplines`), `db/grippers.ts` (`listGripperCatalog`, `rgcInKg` для
конвертации RGC lb→kg). Грип-сила в аналитике строится из `meta` подходов.

**Риски / что чинить.**
- **Дубли кастомных упражнений** (из импорта + ручные) — нет дедупа/мерджа в UI (мерджили
  «натяжку» миграцией вручную, день-32). Нужен пользовательский «об'єднати упражнення».
- Грип-модель живёт в `meta` (JSON) — нет FK, при удалении гриппера из каталога подход осиротеет
  (покажет id без имени). Стоит мягкая защита.
- `meta` — нетипизированный `Record<string,unknown>`; вся семантика (side/cheat/gripper_id/set_type)
  держится конвенцией в коде. Легко сломать молча.

## 9. Аналитика

**По-человечески.** Вкладка «Аналітика»: личные рекорды (вес×повторы и статика-с-весом),
тоннаж (графики/календарь/корреляции), сила хвата (топ-3 по видам установки), восстановление
(OURA-тренды, корреляции с циклом). Всё переключается kg↔lb.

**Под капотом.** `(tabs)/analytics.tsx` + `db/analytics.ts`. Часть метрик считается **в клиенте**
(`analyze()` — рекорды несут и reps[], и time[], чтобы статика не глушила силовые), часть тянется
запросами. Восстановление — из `health_snapshots`; фазы цикла — из `cycle_periods`.

**Риски / ботлнеки / что чинить.**
- ❗**Метрики считаются в клиенте, а `AGENTS.md`/SPEC требуют «метрики в SQL/RPC, ИИ не
  вычисляет».** Правило про ИИ соблюдено (коуч только озвучивает), но «тяжёлая» аналитика на
  клиенте — это расхождение с духом правила и риск производительности на большой истории.
  Кандидат на вынос в SQL-вьюхи/RPC.
- ❗**`personal_records` фактически не наполняется.** Таблица есть, RLS есть, коуч её читает
  (`get_records`) — но **нет триггера/RPC/клиентского кода, который пишет PR при завершении
  тренировки** (в `finishWorkout` и саммари записи PR нет). Аналитика считает рекорды на лету из
  истории, а коуч получает «no records stored yet — derive from history». То есть PR-таблица —
  мёртвый слой. Решение: либо триггер на `sets`, который апсёртит PR, либо убрать таблицу и везде
  считать из истории.
- Клиентский `analyze()` на 30+ тренировках со всеми подходами — потенциально тяжёлый ре-рендер.

## 10. ИИ-коуч (чат)

**По-человечески.** Вкладка «Коуч» — чат. Пишешь вопрос («чи готовий я сьогодні?», «як росте
натяжка?») → коуч отвечает живым текстом, опираясь на твои реальные данные, помнит тебя между
сессиями.

**Под капотом.** `coach-chat/index.ts` — агент с **read-only SQL-инструментами** (get_profile,
get_recent_workouts, get_exercise_history, get_records, get_recovery, get_cycle, get_facts +
remember_fact). Модель сама решает, что запросить → точные свежие числа, без галлюцинаций и без
векторки. Контекст компактный: последние `HISTORY_LIMIT=24` сообщения треда + то, что подтянут
тулзы. Агент-цикл до `MAX_TOOL_TURNS=5`; на последнем турне тулзы не даются (модель обязана
ответить текстом — иначе пустой reply). Персона: живой тон, запрет SWOT-отчётов и training-101,
согласование грамматического рода по полу, plain-text без markdown. Память — `coach_facts`.
Чат — `ai_threads`/`ai_messages`. Каждый вызов модели — через `runIntent` (бюджет/учёт).

**Риски / ботлнеки / что чинить.**
- ❗**Один тред на пользователя** («одно полотно»): `coach-chat` всегда берёт последний
  `ai_threads`. Контекст пухнет (хоть и обрезан 24 сообщ.), нет истории/тем. В плане (START HERE):
  явные треды + кнопка «Нова розмова» + экран истории → заодно дешевле/точнее.
- ❗**Клиент схлопывает ВСЕ серверные ошибки коуча в одну i18n-строку** «Не вдалося зв'язатися
  з коучем» (день-34: пришлось диагностировать временной функцией). Реальная причина (бюджет /
  провайдер / 500) не видна ни юзеру, ни в логе клиента. Стоит различать хотя бы `budget_exceeded`
  vs прочее и логировать код.
- **До 5 вызовов модели на одно сообщение** (агент-цикл) → каждый бьёт `day_request_cap=60`?
  Нет — кап по `requests`, а `ai_record_usage` инкрементит за каждый вызов модели, т.е. одно
  сообщение коуча может «съесть» несколько единиц дневного лимита. На двух юзерах ок, при росте —
  пересмотреть.
- **Глобальный кост-кап (0.3)** кладёт коуча первым делом — что и случилось в дне-34.
- `remember_fact` не дедупит — коуч может насохранять похожих фактов; нет TTL/ревизии `coach_facts`.

## 11. Голосовой ввод коуча (STT)

**По-человечески.** В чате коуча — микрофон → записываешь голос → расшифровка падает в поле
ввода (правишь и отправляешь сам). TTS нет. На вебе микрофон скрыт.

**Под капотом.** `coach.tsx` + `db/coach.ts`: `expo-audio` запись → base64 → Edge `transcribe`
(OpenAI `gpt-4o-mini-transcribe`, language-хинт из профиля). Аудио нигде не хранится; ключ
серверный. Авто-стоп 20с (`MAX_RECORDING_SEC`), отмена ✕, cleanup при уходе с вкладки.
День-34: кост теперь **по размеру файла** (`bytes/16000`), `MAX_BYTES=8МБ` → жёсткий потолок
~$0.026/запрос (раньше биллился по клиентскому `durationSec` → и взорвался).

**Риски / что чинить.**
- **Мёртвый параметр `durationSec`** ещё шлётся клиентом (`coach.tsx`/`transcribeAudio`) — сервер
  игнорит, можно убрать (хвост в START HERE).
- **STT на web нет** (микрофон скрыт) — план: Web Speech API.
- Премиум-лимит записи >20с (константа) — задел под монетизацию (раздел 16).

## 12. OURA (восстановление)

**По-человечески.** «Здоров'я» → вставляешь OURA Personal Access Token → «Підключити» →
синк за год → карточки (readiness/sleep/HRV/RHR/temp/…), тап по карточке = справка. «Синхр.».

**Под капотом.** `health.tsx` → `oura-connect` (валидирует токен запросом к OURA, кладёт в
`private.oura_tokens` через security-definer RPC `store_oura_token`, **токен на клиент не
возвращается**) → `oura-sync` (тянет 365 дн, апсёрт в `health_snapshots`, не удаляет). Коуч и
аналитика читают `health_snapshots`. Возраст последнего снимка отслеживается (stale-метка, и коуч
обязан учитывать свежесть).

**Риски / что чинить.**
- ❗**Только PAT, нет OAuth.** Это «и то по пат токену» из твоего текста. Для обычного юзера
  «вставь Personal Access Token» — нерабочий UX. Под публику нужен OURA OAuth flow.
- PAT хранится в `private.oura_tokens` (вне RLS-схемы, доступ через RPC) — хорошо, но проверь,
  что схема `private` действительно недоступна `anon/authenticated` и токен не в открытом виде в
  логах Edge.
- Синк ручной (кнопка) — нет авто-обновления (cron). Коуч про свежесть честно говорит, но данные
  устаревают, если юзер не жмёт «Синхр.». Кандидат на scheduled-функцию.
- Только OURA. Apple Health/Google Fit/Garmin — задел на будущее (на экране есть «soon»-блок).

## 17. Главное: как из «вонючего Expo/Metro» сделать настоящие APK/IPA

Сейчас запуск = `expo start` + Metro + Expo Go в одной Wi-Fi сети с телефоном (порт пинён 8090,
см. `.claude/launch.json`). Это дев-режим, не приложение. Чтобы получить устанавливаемые
**Android (.apk/.aab)** и **iOS (.ipa)** — путь через **EAS Build** (облачная сборка Expo).
`SPEC.md` это и предполагает (EAS Build + Submit), но ❗**`eas.json` в репозитории нет** — то
есть пайплайн ещё не заведён.

### Что это даёт сразу
- **Android dev/preview build** — ставишь .apk на свой S25, **Metro не нужен**, приложение живёт
  само (но интернет всё ещё нужен — см. 0.1). Это закрывает твою боль «не могу вести тренировки
  без разворачивания шляпы». iOS-таргет тоже соберётся, но для установки на iPhone нужен Apple
  Developer-аккаунт ($99/год) и регистрация устройства (ad-hoc) либо TestFlight.

### Пошагово (предлагаемый план, не выполняю без твоего ОК)
1. `npm i -g eas-cli` → `eas login` (твой Expo-аккаунт).
2. `eas build:configure` — создаст `eas.json` с профилями. Завести три:
   - `development` (dev-client, internal distribution) — для отладки на устройстве;
   - `preview` (internal, Android `.apk`) — «просто пользоваться» здесь и сейчас;
   - `production` (Android `.aab`, iOS store) — для публикации.
3. **Секреты/ENV:** сейчас клиент читает `EXPO_PUBLIC_SUPABASE_URL`/`ANON_KEY` из `.env`. Для EAS
   их надо положить в **EAS Environment Variables** (или `eas secret`) на каждый профиль — иначе
   сборка уйдёт без бэкенда. Это must-fix перед первой сборкой.
4. **iOS-специфика:** `app.json.ios` сейчас почти пустой — нет `bundleIdentifier`. Нужны
   `ios.bundleIdentifier` и `android.package` (обязательны для сборки/публикации). Apple-аккаунт
   для подписи; EAS умеет управлять сертификатами сам.
5. **Разрешения:** микрофон уже описан (`expo-audio.microphonePermission`). Проверить, что описаны
   все нужные usage-strings для iOS перед сабмитом.
6. `eas build -p android --profile preview` → получаешь ссылку на `.apk` → ставишь на S25.
   Дальше `eas build -p ios` / `eas submit` когда дойдёт до сторов.

### Что «починить заодно» при выходе из дев-режима (мой список приоритетов)
Высокий (блокеры реального использования/релиза):
1. **Решить вопрос offline (0.1)** — хотя бы кэш чтения + очередь записи подходов. Без этого
   «настоящее приложение» в зале всё равно будет подводить.
2. **Per-user бюджет ИИ (0.3/16)** — иначе один баг/юзер кладёт ИИ всем (уже было).
3. **`eas.json` + ENV в EAS + bundleId/package** — без этого сборки нет.
4. **Удаление аккаунта + приватность/условия** — требования сторов.
5. **Восстановление пароля + (для iOS) Sign in with Apple** при наличии стороннего логина.

Средний (качество/долги):
6. **`personal_records` или наполнять, или удалить** (сейчас мёртвый слой).
7. **Привести `AGENTS.md`/`SPEC.md` к реальности** про sqlite/offline (или сделать, или убрать).
8. **Различать ошибки коуча** на клиенте (не одна строка на всё).
9. **Optimistic updates / autosave** в тренировке (потеря ввода, перезагрузки дерева).
10. **OURA OAuth** вместо PAT; авто-синк (cron-функция).
11. **Дедуп кастомных упражнений**; защита `meta.gripper_id` от осиротения.
12. **Вынести тяжёлую аналитику в SQL/RPC** (соответствие правилу + перф).

Низкий (косметика):
13. Нумерация тренировок `#N` при >30; дискаверабилити «імпорт минулого» и «старт з нуля»;
    онбординг-CTA к импорту; сбор `bodyweight`/`height` в онбординге.
